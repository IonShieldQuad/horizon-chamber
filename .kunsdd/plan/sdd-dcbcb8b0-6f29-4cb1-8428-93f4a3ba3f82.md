# Implementation Plan: Dynamic Kanban Goals System (v0.3)

## 1. Summary

Replace the simple chaos-classification-card system with a full dynamic kanban goal board. Introduce three goal types (long\_term, habit, maintenance), a daily task generation engine, carry-over management, AI-powered task splitting and frequency suggestions, performance analysis with adaptive nudges, and a 4-column kanban frontend — all within the existing FastAPI + SQLite + vanilla JS architecture.

**Design north star:** The board must feel as low-friction as a physical kanban board with sticky notes. Every interaction takes 1–2 actions maximum. No modals, no required fields, no confirmation dialogs for common operations.

---

## 2. Pre-mortem Analysis

### 2.1 Risk Inventory

| Risk | Type | Classification | Mitigation |
|------|------|----------------|------------|
| DB migration (v2→v3) corrupts existing chaos/activity data | **Tiger** | **Launch-Blocking** | Run migration inside a transaction; test rollback; keep old table schemas intact; add pre-migration integrity check |
| Goal engine generates duplicate tasks on rapid board opens | **Tiger** | **Launch-Blocking** | Guard generation with a date-level "already generated" flag in memory + DB; deduplicate by (goal_id, date) unique constraint or upsert |
| AI reliance on DeepSeek creates a hard failure mode when key is missing | **Tiger** | **Launch-Blocking** | Graceful degradation specified in §3.5 of this plan; every AI feature has a non-AI fallback path |
| Frontend drag-and-drop on mobile/touch is janky with native HTML5 drag | Paper Tiger | Fast-Follow | Start with click-to-move and simple swipe-to-skip; native HTML5 drag for desktop; touch polish deferred to fast-follow |
| Performance analysis nudges become annoying and users disable them entirely | Paper Tiger | Track | Nudge priority system (§5.2) limits frequency; one-line passive display never popups; env-var kill switch |
| Background daily generation conflicts with FastAPI async model (database contention) | Tiger | **Launch-Blocking** | Use a single dedicated aiosqlite connection for the engine loop; acquire a named lock/mutex around generation; test with concurrent board loads |
| Large existing databases (>1000 activity rows) cause slow analysis queries | Elephant | Investigate | Add indexes on activity_log.timestamp and task_log.date; measure with 10k rows before shipping |
| User has overlapping goal/task data from chaos table — migration semantics unclear | Elephant | Investigate | Chaos table remains as inflow channel; no data migration from chaos to goals needed — users create goals fresh via the new UI |

### 2.2 Launch-Blocking Tiger Mitigations

| Tiger | Owner | Decision Date | Concrete Action |
|-------|-------|---------------|-----------------|
| DB migration corruption | Developer | Before Phase 1 starts | Write migration SQL with `BEGIN TRANSACTION`, verify PRAGMA integrity_check before and after, keep a backup of the DB file path |
| Duplicate task generation | Developer | Before Phase 2 ships | Add `UNIQUE(goal_id, date)` on tasks table (or use INSERT OR IGNORE); add in-memory `_generated_dates: set[str]` reset at midnight |
| AI key missing breaks everything | Developer | Before Phase 3 ships | Every AI route returns a 200 with `"ai_disabled": true` and a fallback body when key is missing; frontend checks this field |
| DB contention from background engine | Developer | Before Phase 2 ships | Provide an optional `engine_db_path` env var; default to sharing main DB; use aiosqlite with WAL mode (already set) and `BEGIN IMMEDIATE` for writes |

### Phase 2: Goal Engine (2–3 hours)

#### Step 2.1: Create `goal_engine.py` — config and constants (covers: goal engine configuration)

```python
# goal_engine.py

import asyncio
import logging
import os
from datetime import date, datetime, timedelta
from typing import Optional

import db as db_mod

logger = logging.getLogger(__name__)

GOAL_REMINDER_GAP_DAYS = int(os.getenv("GOAL_REMINDER_GAP_DAYS", "3"))
GOAL_MAX_CARRY_OVER = int(os.getenv("GOAL_MAX_CARRY_OVER", "3"))
GOAL_CLEANUP_TIME = os.getenv("GOAL_CLEANUP_TIME", "04:00")
GOAL_GENERATE_ON_OPEN = os.getenv("GOAL_GENERATE_ON_OPEN", "true").lower() in ("true", "1")

_generated_today: set[str] = set()  # track (goal_id, date) to prevent double-gen
_engine_running = False
_last_run: Optional[str] = None
_last_cleanup: Optional[str] = None
```

#### Step 2.2: Implement daily generation logic (covers: long_term task generation, habit task generation, maintenance task generation)

- `async def _generate_for_long_term(goal: dict, today_str: str) -> int`:
  - Query last task for this goal
  - If no task exists in last N days → create reminder task (title = goal.title + " — next step?")
  - If last task was `done` → create next occurrence
  - If last task was `skipped` or `carried_over` → copy title, increment carry_over_count
  - If carry_over_count > GOAL_MAX_CARRY_OVER → flag (log entry, nudge trigger)
- `async def _generate_for_habit(goal: dict, today_str: str) -> int`:
  - If no task exists for today for this goal → create one
- `async def _generate_for_maintenance(goal: dict, today_str: str) -> int`:
  - Compute interval: if `frequency == 'custom'` use `custom_interval_days`, else 1 for daily, 7 for weekly
  - If today >= last_task_date + interval → create task

#### Step 2.3: Implement cleanup routine (covers: cleanup incomplete habit/maintenance tasks)

- `async def _cleanup_yesterday(yesterday_str: str)`:
  - For all incomplete habit tasks with yesterday's date → set status to 'skipped', log to task_log
  - For all incomplete maintenance tasks with yesterday's date → set status to 'skipped', log to task_log
  - Long-term tasks are NOT cleaned (they carry over)

#### Step 2.4: Implement main generation engine (covers: daily generation trigger, generate_today_tasks)

- `async def generate_today_tasks(today_str: str | None = None) -> dict`:
  - If today_str is None, use today's date
  - Get all non-archived, non-paused goals
  - For each goal, call the appropriate generator
  - Skip if already generated today (check _generated_today)
  - Return `{"created": int, "skipped": int, "carried_over": int, "goals_processed": int}`
- `async def start_engine()`:
  - Background loop that runs cleanup at GOAL_CLEANUP_TIME
  - Periodically checks if today's generation is needed (lazy trigger)
  - Sleep 60s between checks
- Editable `_generated_today` tracking: reset at midnight

#### Step 2.5: Implement board assembly (covers: get_board endpoint)

- `async def get_board(date_str: str | None = None) -> dict`:
  - Default to today's date
  - If lazy generation enabled and today not yet generated → trigger generation
  - Query tasks for the given date grouped by status
  - Assemble 4 columns: `today` (pending + doing), `in_progress` (doing with carry_over), `overdue` (pending tasks with date < today), `done`
  - Include goal metadata and per-goal stats
  - Return `{"columns": {...}, "goals": [...], "stats": {...}}`

#### Step 2.6: Implement analysis engine (covers: performance signal detection, analysis endpoint)

- `async def compute_analysis(date_str: str | None = None) -> dict`:
  - Compute completion_rate = done / (done + skipped) for the day/week
  - Compute carry_over_depth = MAX(carry_over_count) for active tasks
  - Compute carry_over_breadth = COUNT(carried_over) / COUNT(pending)
  - Compute stagnation_days for each long_term goal
  - Compute skip_pattern: consecutive skips per goal
  - Return all signals as a structured dict
- `async def get_goal_analysis(goal_id: int) -> dict`:
  - Per-goal: progress_pct, days_active, tasks_done, tasks_skipped, trend

#### Step 2.7: Implement nudge system (covers: nudge generation, nudge priority)

- `async def generate_nudges(analysis: dict) -> list[dict]`:
  - Map signals to nudge levels: Info (completion_rate > 85% or < 50%), Suggestion (carry_over_depth > 3), Alert (carry_over_breadth > 50%)
  - Generate human-readable nudge text (non-judgmental, actionable)
  - Respect max frequency per level
  - Return ordered list: `[{"level": "info", "text": "...", "action": "...", "dismissable": true}]`
- If `NUDGES_ENABLED` is false, return empty list
- Nudges are logged to task_log for transparency

#### Step 2.8: Write tests (`tests/test_goal_engine.py`) (covers: engine test coverage)

- `test_generate_long_term_first_task`: no existing tasks → creates one reminder
- `test_generate_long_term_carry_over`: skipped task → same title, carry_over_count > 0
- `test_generate_long_term_max_carry_over`: exceeds threshold → generates nudge trigger
- `test_generate_habit_daily`: no task today → creates one
- `test_generate_maintenance_interval`: respects custom_interval_days
- `test_cleanup_habits`: yesterday's incomplete habit → marked skipped
- `test_cleanup_maintenance`: yesterday's incomplete maintenance → marked skipped

### Phase 3: API Endpoints (1–2 hours)

#### Step 3.1: Goal CRUD routes in `main.py` (covers: goal CRUD API)

- `GET /api/goals` — list goals, supports `?type=long_term&archived=true`
- `POST /api/goals` — create goal, body: `{title, type, frequency?, custom_interval_days?, target_days?, description?}`. If AI key available and no `type` provided, call `deepseek_client.classify_as_goal(title)` for automatic type detection. Returns the created goal.
- `GET /api/goals/{id}` — single goal with progress data (calls `goal_engine.get_goal_progress`)
- `PUT /api/goals/{id}` — update goal fields
- `DELETE /api/goals/{id}` — soft-delete (set archived=1)

#### Step 3.2: Task management routes (covers: task update, task split, board endpoint)

- `GET /api/board` — full kanban board state for today (calls `goal_engine.get_board`)
- `GET /api/board?date=2025-01-15` — board for specific date
- `PATCH /api/tasks/{id}` — update task: `{status: "done"|"doing"|"skipped"|"pending"}`. When set to "done", also sets `completed_at` and inserts task_log entry.
- `POST /api/tasks/{id}/split` — AI-split task into subtasks (calls DeepSeek, creates new tasks with parent_task_id)
- `POST /api/tasks/{id}/suggest` — AI-suggest next step/frequency for parent goal

#### Step 3.3: Analysis and engine control routes (covers: analysis API, engine control API)

- `GET /api/analysis` — return performance signals (calls `goal_engine.compute_analysis`)
- `GET /api/analysis/goal/{id}` — per-goal analysis
- `POST /api/analysis/refresh` — force re-run analysis
- `POST /api/engine/generate` — manually trigger daily generation
- `GET /api/engine/status` — engine status: `{"running": bool, "last_run": str, "last_cleanup": str, "today_generated": bool}`

#### Step 3.4: Update `/api/today` (covers: today list API — breaking change)

- `GET /api/today` now returns priority-ordered task list: `{"tasks": [{"id", "goal_id", "title", "status", "date", "goal_type", "carry_over_count"}], "date": "2025-01-15"}`

#### Step 3.5: Start goal engine in lifespan (covers: engine startup)

- In `main.py`'s `lifespan` function, after existing startup code: `engine_task = asyncio.create_task(goal_engine.start_engine())`
- Add shutdown: `engine_task.cancel()` in the cleanup block
- Import `goal_engine` module

#### Step 3.6: Add request schemas (covers: request validation)

Add Pydantic models at the top of `main.py`:
- `CreateGoalRequest(BaseModel)` — title, type (optional), frequency, custom_interval_days, target_days, description
- `UpdateGoalRequest(BaseModel)` — all fields optional
- `UpdateTaskRequest(BaseModel)` — status, description, sort_order
- `EngineGenerateResponse(BaseModel)` — for typed responses

#### Step 3.7: Write tests (`tests/test_goals_api.py`) (covers: API test coverage)

## 7. Risks and Dependencies

- **External dependency:** DeepSeek API availability. All AI features degrade gracefully when the API key is missing or the service is down.
- **Database lock contention:** The engine loop and API handlers both access the same SQLite database. Mitigated by WAL mode (already set) and `BEGIN IMMEDIATE` for write transactions.
- **Frontend complexity:** Vanilla JS kanban with drag-and-drop, swipe, and SSE is non-trivial. Start with click-to-move and native HTML5 drag; add swipe polish after core is working.
- **Test isolation:** All tests use `tmp_path` databases to avoid cross-contamination. Engine tests mock `datetime` for deterministic date behavior.
